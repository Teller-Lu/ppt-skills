# 保真与内容适配方法论

> 起因:首套 round-trip 导出后,用户逐页发现一批问题,要求**从方法论上根治**——换模板、填真实内容都不再复发。
> 结论先行:**那次是"一比一回环复刻"测试,而 skill 根本不走这条回环**;但回环暴露的问题正好指明了 skill 必须内建的能力。

---

## 一、先厘清:回环测试 ≠ skill 的工作方式(最重要)

| | 这次测试(回环) | skill 真正的路径 |
|---|---|---|
| 流程 | 你的 pptx → 解析成 SVG → **再导回** pptx | 解析模板**一次**提炼风格/组件 → **AI 重新手写干净 SVG(你的内容)** → 导出 |
| 目的 | 验证引擎 + 出张去章副本 | 用你的审美 + 你的内容做**新** PPT |
| 你看到的瑕疵 | 大多是**解析器没吃透原 pptx 高级效果**的转换损耗 | 由 AI 按引擎安全方式重新表达,不继承这些损耗 |

→ 所以**不是"我复刻失败了"**,而是"回环这条路本来就会丢东西";而且它**帮我们提前看清了两件事**:① 引擎到底能不能可靠表达某种效果;② 真实内容塞进模板会怎样。

## 三、引擎能/不能(据 ppt-master `shared-standards.md`,这是设计底线)

**能可靠表达(放心用)**:渐变填充(线/面)`gradFill`、阴影 `feGaussianBlur+feOffset→outerShdw`、发光 `glow`、`<image>` 上的非矩形裁剪(圆/圆角/多边形/路径)、连接箭头 `marker-end`、虚线/线连接、旋转、圆环/扇形(手算 arc path)、OOXML 预设图案。

**不能可靠表达(必须改写或烘焙)**:
- `<mask>`、组 `opacity`、`<image opacity>`、`rgba()`、`<style>/class`、`symbol+use`、`textPath`、`@font-face`、`foreignObject`。
- ⚠️ **文字图片填充 / 文字镂空图(text-knockout image fill)= 没有 PPT 路径** → 官方结论:**烘焙进图片**。p3 的"01"正属此类。
- 嵌入图表对象 → 解析只得占位符 → 须自己用 SVG/真实绘图重建。

> 所以"保留你模板里的高级技法"不是靠复刻,而是靠**把技法翻译成引擎安全的 SVG 写法,个别翻不了的就烘焙成图**。

---

## 四、方法论 A — 组件是"蒸馏"出来的,不是原样搬运

提炼组件(阶段 4)**不是**把 `pptx_to_svg` 的原始 SVG 存下来当组件,而是把每页**重新改写**成:

1. **只用引擎安全构造**(§三白名单)。原模板的效果按"技法→引擎安全写法"映射重建:
   | 模板技法 | 引擎安全实现 |
   |---|---|
   | 卡片阴影 | `filter`(feGaussianBlur+feOffset) |
   | 渐变线/面、波形渐变 | `linearGradient`/`radialGradient` 作 stroke/fill |
   | 圆环、装饰环线 | `<circle fill=none stroke>` |
   | 扇形圆环、甜甜圈 | 手算 arc `<path>` |
   | 水滴/六边形等异形 | `<path>`/`<polygon>`(或异形裁剪图) |
   | **图片填充文字(如"01")** | **烘焙成 PNG**,组件用 `<image>` 引用(序号 01–10 可预烘焙一套) |
   | 透明叠色 | `fill-opacity` / 叠加渐变 `<rect>` |
2. **保留技法,不丢效果**:维护一个小"技法库",每个签名效果都有对应的引擎安全写法,绝不"因为难就删掉"。
3. **构造兼容门**(新增校验):扫描组件/成品 SVG 里有没有"引擎不可靠"的构造 → 提前报错。杜绝"模板里好看、导出就崩"。

---

## 五、方法论 B — 内容适配(根治 p10/p15/p20 这类)

**前提**:你给的真实内容,长度/条数几乎一定和模板不一样。所以**绝不能**"按模板坐标把字塞进去"。

1. **组件带"内容契约"(content contract),不是固定文字**。每个文字槽声明:
   - `role`(标题/正文/标签…)、`max_lines`、`min/max font`、**`content_box`(隐形内容边界 = 设计上不许越过的框,通常比可见块小)**、`overflow`(溢出策略:`wrap → shrink → grow-block → relayout`)。
   - 这些**从模板量出来**(如 p10 蓝块的内边界、p15/p20 两侧的隐形边界)。
2. **内容适配 pass(自动回流)**:填充时按契约执行——先折行 → 不够再缩字(到校准下限)→ 再不够扩块/挪邻居 → 仍不行则**换更密的组件或拆页**。p10 的两行折行就是这一步自动完成。
3. **Strategist 上游路由**:排版前先估每页内容体量,**挑容量匹配的组件**(4 条要点→`seg-4` 而非硬塞 `seg-3`),从源头避免溢出,而不是事后补救。

---

## 六、方法论 C — 双门把关(把"不许越界"变成可执行)

1. **美学质量门**(C,已设计)新增/强调:
   - **隐形边界溢出**:文本/块越过组件声明的 `content_box`(不只是画布边界)→ `error`。
   - 元素碰撞/重叠(p16 文字压图案)、对比度、对齐、字号下限。
2. **构造兼容门**(§四.3):引擎不可靠构造 → `error`。
3. 严重性沿用 ppt-master:`error` 阻塞 → **定向重写该页**(带内容契约重新回流),非机械 patch。

---

## 七、一句话原则 + 写入位置

> **组件 = 引擎安全 + 带内容契约 + 可回流的参数化模板;AI 必须"按真实内容重新手写并自适应排版",再由"美学门 + 构造门"把关。绝不是"复印模板换文字"。**

这条要写进:
- `SKILL.md`(编排:选组件→按内容契约回流→双门校验);
- `references/executor-*.md`(手写纪律:引擎安全构造、隐形边界、折行缩字);
- `素材模型与组件库规格.md`(组件 schema 增 `content_contract` + `technique_map` + 烘焙资产);
- `美学质量门规格.md`(增 隐形边界溢出 + 构造兼容)。

→ 这样**换任何模板、填任何真实内容**,走的都是同一套"蒸馏组件 + 内容回流 + 双门"机制,不再复发本次问题。
