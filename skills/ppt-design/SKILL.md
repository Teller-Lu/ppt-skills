# PPT Design · 设计前脑

> 做 PPT 的**第一步**：开工先想清楚**每页要表达什么核心**，而不是先翻素材库套模板。
> 本 skill 只产出一份**《逐页设计稿》**；经用户审批后，交给 `ppt-skills` 制作 + 美化成可编辑 pptx。
> **本 skill 绝不碰** builder / 素材库 / pptx / 渲染——那是 `ppt-skills` 的活。设计阶段手边没有"锤子"，正是为了逼自己先想"表达"。

**核心流水线**：`读材料 → 提炼 3 条主线 → 逐页四问(一句话 / 表达式 / 为什么 / 真实素材) → 汇成《逐页设计稿》 → 交用户审 → 批准后交棒 ppt-skills`

---

> [!CAUTION]
> ## 🚨 交接协议（最高优先级，违反即失败）
>
> 1. **只设计、不制作** —— 本 skill 的产物**只有一份《逐页设计稿》md**。绝不写 pptx、绝不调 builder / 素材、绝不渲染。"想动手做了"就是越界，停。
> 2. **设计稿是唯一交接物，且必须经用户审批** —— 没有用户点头的设计稿，不许进 `ppt-skills`；`ppt-skills` 入口有前置门，会把"无批准设计稿"的请求弹回这里。
> 3. **表达优先、不素材优先** —— 每页先定"用什么表达"（查《表达式词典》），再谈好不好看。**绝不能因为"模板 / 库里有这个件"就选它**。
> 4. **真实素材不编造** —— 缺真实图 / 数据的页，设计稿里标 **【需提供】** 请用户补，绝不用 AI 生成 / 假数据顶替。

## 主流程

### 第 1 步 · 读材料 + 定基调
读全部源材料（文档 / 数据 / 已填《资料采集清单》）。锁定基调：受众 / 场合 / 时长 / 目标页数。

### 第 2 步 · 提炼 3 条主线
从"受众最该记住的结论"倒推出 **≤3 条主线**，全篇围绕它们铺。主线不清，先找用户对齐，**别急着分页**。

### 第 3 步 · 逐页四问（核心，缺一不可）
每一页强制回答：

1. **① 一句话**：这页要让听众记住的**唯一核心**（takeaway）。写不出一句话 = 这页还没想清楚，或该并页。
2. **② 表达式**：用什么形式承载？**查《表达式词典》** 指到具体一类（流程 / 层级 / 计划 / 对比 / 数据 / 证据 / 概念 / 并列要点…）。
3. **③ 为什么**：为什么这个表达式最清楚？**对照词典的反模式红线自证**——禁止理由"模板里有这个件"。
4. **④ 真实素材**：需要哪些真实图 / 数据？缺的标 **【需提供】**。

### 第 4 步 · 汇成《逐页设计稿》
用 `references/逐页设计稿模板.md` 把全局块（受众 / 场合 / 时长 / 页数 / 3 主线）+ 逐页四问汇成一份 md。

### 第 5 步 · 表达层自检（出稿前过一遍）
- 这页核心**一眼可见**吗？
- 表达式是这页内容的**最优解**吗（不是"模板顺手"）？
- 有没有**该用真图却手画**、**该 mermaid 却框图**？
- 有没有**装饰性占位 / 浪费空间**？

任一不过 → 回第 3 步重想，**别带病交稿**。

### 第 6 步 · 交用户审 → 交棒
设计稿交用户审批。**批准后**才交给 `ppt-skills` 制作。

## mermaid 工作流（结构类首选，别手搓）

结构 / 流程 / 甘特 / 层级这类，**首选 mermaid 或用户真实图**，手搓矢量是最后手段：
**我写 mermaid 代码 → 用户用 Obsidian Mermaid Exporter 插件导成 PNG/SVG → 交 `ppt-skills` 当真实图嵌入**（可设 mermaid 主题色贴近品牌蓝）。

分寸：mermaid 静态、风格通用，**不全用**——封面 / 结尾 / 并列枢纽 / 对比 / 数据这些"品牌时刻"仍交 `ppt-skills` 用库件；mermaid 专治**手搓易翻车的结构图**。详见《表达式词典》。
